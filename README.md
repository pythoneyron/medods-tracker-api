# Medods Tracker API

API-only приложение на Ruby on Rails для модуля трекера рабочих задач в медицинской информационной системе. Сервис позволяет персоналу регистрироваться, авторизоваться по JWT, вести личные задачи, фильтровать их по статусу и дате, помечать задачи пользовательскими и системными тегами, а также работать с повторяющимися задачами.

Повторяющиеся задачи реализованы без предварительного создания записей на годы вперёд: в базе хранится одна задача-серия с правилом повторения, а конкретные экземпляры генерируются только для запрошенного временного окна. Индивидуальный статус конкретного экземпляра хранится отдельно в `task_occurrences`.

## Технологии

- Ruby `4.0.1`
- Ruby on Rails `8.1.3` в режиме API-only
- PostgreSQL
- Puma
- Jbuilder для JSON-ответов
- Devise и `devise-jwt` для регистрации, входа, выхода и JWT-аутентификации
- `JwtDenylist` для ревокации JWT при выходе
- Pagy для пагинации
- RSpec, FactoryBot, Shoulda Matchers и Faker для тестов
- rswag для OpenAPI/Swagger-документации
- RuboCop, Brakeman и Bundler Audit для статических проверок
- Docker и Docker Compose для контейнерного запуска

## Реализованный функционал

### Аутентификация

Все пользовательские API-эндпоинты находятся под префиксом `/api/v1`.

- `POST /api/v1/users` - регистрация пользователя.
- `POST /api/v1/users/sign_in` - вход пользователя.
- `DELETE /api/v1/users/sign_out` - выход пользователя и отзыв текущего JWT.
- JWT возвращается в заголовке `Authorization`.
- Защищенные эндпоинты требуют заголовок `Authorization: Bearer <token>`.

### Задачи

Реализован базовый CRUD задач:

- `GET /api/v1/tasks` - список задач текущего пользователя.
- `GET /api/v1/tasks/:id` - просмотр одной задачи текущего пользователя.
- `POST /api/v1/tasks` - создание задачи для текущего пользователя.
- `PATCH /api/v1/tasks/:id` - обновление задачи текущего пользователя.
- `DELETE /api/v1/tasks/:id` - удаление задачи текущего пользователя.

Задача содержит:

- `title` - название.
- `description` - описание.
- `due_date` - дата выполнения.
- `status` - статус.
- `tags` - связанные теги.

Поддерживаемые статусы:

- `planned`
- `pending`
- `in_progress`
- `done`
- `cancelled`

Для списка задач реализованы фильтры:

- `status` - фильтр по статусу.
- `date` - фильтр по точной дате выполнения.
- `date_from` и `date_to` - фильтр по диапазону дат.

Задачи изолированы по пользователям: пользователь не может получить, изменить или удалить чужую задачу.

### Повторяющиеся задачи

Сервис поддерживает повторяющиеся задачи. Повторяющаяся задача хранится как одна запись в таблице `tasks` с настройками периодичности. Система не создаёт заранее отдельные записи задач на будущие даты.

При запросе списка задач за период API генерирует виртуальные экземпляры повторяющихся задач только внутри запрошенного временного окна.

Поддерживаемые типы повторения:

- `none` - обычная задача без повторения.
- `daily` - ежедневная задача, каждый n-й день.
- `monthly_day` - ежемесячная задача на конкретное число месяца от 1 до 31.
- `specific_dates` - задача только на конкретные даты.
- `even_days` - задача по чётным числам месяца.
- `odd_days` - задача по нечётным числам месяца.

Поля периодичности:

- `recurrence_type` - тип повторения.
- `recurrence_config` - настройки повторения в формате JSON.
- `recurrence_starts_on` - дата начала повторения.
- `recurrence_ends_on` - дата окончания повторения. Может быть `null`, если серия не имеет даты окончания.

Для повторяющихся задач поле `due_date` используется как базовая дата задачи. Если `recurrence_starts_on` не передан, он автоматически устанавливается равным `due_date`.

Для `GET /api/v1/tasks` API возвращает календарную выдачу задач. У каждого элемента есть поле `occurrence_date`, которое показывает конкретную дату экземпляра задачи.

Пример ответа для повторяющейся задачи:

```json
{
  "id": 1,
  "task_id": 1,
  "occurrence_date": "2026-05-11",
  "title": "Daily patient round",
  "description": "Check patients status",
  "due_date": "2026-05-10",
  "status": "planned",
  "recurring": true,
  "recurrence_type": "daily",
  "recurrence_config": {
    "interval": 1
  },
  "recurrence_starts_on": "2026-05-10",
  "recurrence_ends_on": null,
  "tags": [],
  "created_at": "2026-05-10T10:00:00Z",
  "updated_at": "2026-05-10T10:00:00Z"
}
```

#### Проблема бесконечности

Если задача повторяется без даты окончания, например ежедневный обход пациентов, система не создаёт миллионы записей в базе данных.

Вместо этого:

1. В `tasks` хранится одна задача-серия.
2. В `recurrence_type`, `recurrence_config`, `recurrence_starts_on` и `recurrence_ends_on` хранится правило повторения.
3. При запросе списка задач сервис генерирует экземпляры только внутри запрошенного диапазона дат.
4. Максимальный диапазон запроса ограничен `366` днями.

Например:

```http
GET /api/v1/tasks?date_from=2026-05-10&date_to=2026-05-12
```

Для ежедневной задачи вернёт только три экземпляра:

```text
2026-05-10
2026-05-11
2026-05-12
```

Новые записи в таблице `tasks` при этом не создаются.

#### Независимый статус конкретного экземпляра

Статус всей серии хранится в `tasks.status`.

Если пользователь меняет статус конкретного экземпляра повторяющейся задачи, создаётся или обновляется запись в таблице `task_occurrences`.

Таблица `task_occurrences` хранит индивидуальное состояние конкретного экземпляра:

- `task_id`
- `occurrence_date`
- `status`

Для пары `task_id + occurrence_date` используется уникальный индекс, чтобы у одного экземпляра задачи было только одно состояние.

Пример:

```text
task_id: 1
occurrence_date: 2026-05-11
status: done
```

Это означает, что только экземпляр за `2026-05-11` выполнен. Экземпляры за `2026-05-10` и `2026-05-12` останутся в базовом статусе серии.

Endpoint для обновления статуса конкретного экземпляра:

```http
PATCH /api/v1/tasks/:task_id/occurrences/:date
```

Пример:

```http
PATCH /api/v1/tasks/1/occurrences/2026-05-11
```

Body:

```json
{
  "task_occurrence": {
    "status": "done"
  }
}
```

После этого `GET /api/v1/tasks?date_from=2026-05-10&date_to=2026-05-12` вернёт:

```json
{
  "tasks": [
    {
      "task_id": 1,
      "occurrence_date": "2026-05-10",
      "status": "planned"
    },
    {
      "task_id": 1,
      "occurrence_date": "2026-05-11",
      "status": "done"
    },
    {
      "task_id": 1,
      "occurrence_date": "2026-05-12",
      "status": "planned"
    }
  ]
}
```

Таким образом, изменение одного экземпляра не меняет статус всей серии.

#### Примеры `recurrence_config`

##### Обычная задача без повторения

```json
{
  "task": {
    "title": "Call patient",
    "description": "Clarify patient status",
    "due_date": "2026-05-10",
    "status": "planned",
    "recurrence_type": "none",
    "recurrence_config": {}
  }
}
```

##### Ежедневная задача каждый день

```json
{
  "task": {
    "title": "Daily patient round",
    "description": "Check patients status",
    "due_date": "2026-05-10",
    "status": "planned",
    "recurrence_type": "daily",
    "recurrence_config": {
      "interval": 1
    },
    "recurrence_starts_on": "2026-05-10",
    "recurrence_ends_on": null
  }
}
```

##### Ежедневная задача каждый второй день

```json
{
  "task": {
    "title": "Patient calls every second day",
    "description": "Call patients according to schedule",
    "due_date": "2026-05-10",
    "status": "planned",
    "recurrence_type": "daily",
    "recurrence_config": {
      "interval": 2
    },
    "recurrence_starts_on": "2026-05-10",
    "recurrence_ends_on": null
  }
}
```

Если `recurrence_starts_on` равен `2026-05-10`, то задача будет выпадать:

```text
2026-05-10
2026-05-12
2026-05-14
2026-05-16
```

##### Ежемесячная задача на конкретное число месяца

```json
{
  "task": {
    "title": "Prepare monthly report",
    "description": "Prepare medical department report",
    "due_date": "2026-05-15",
    "status": "planned",
    "recurrence_type": "monthly_day",
    "recurrence_config": {
      "day": 15
    },
    "recurrence_starts_on": "2026-05-01",
    "recurrence_ends_on": null
  }
}
```

Если указано число `31`, то месяцы без 31-го числа пропускаются.

Например для периода с января по апрель задача выпадет:

```text
2026-01-31
2026-03-31
```

Февраль и апрель будут пропущены.

##### Задача на конкретные даты

```json
{
  "task": {
    "title": "Scheduled operations",
    "description": "Operations on selected dates",
    "due_date": "2026-05-10",
    "status": "planned",
    "recurrence_type": "specific_dates",
    "recurrence_config": {
      "dates": [
        "2026-05-10",
        "2026-05-17",
        "2026-05-24"
      ]
    },
    "recurrence_starts_on": "2026-05-10",
    "recurrence_ends_on": null
  }
}
```

```json
    "recurrence_config": {},
    "recurrence_starts_on": "2026-05-02",
    "recurrence_ends_on": null
  }
```

##### Задача по нечётным числам месяца

```json

    "due_date": "2026-05-01",
```

### Теги

Реализован CRUD тегов:

- `GET /api/v1/tags` - список доступных тегов.
- `GET /api/v1/tags/:id` - просмотр тега.
- `POST /api/v1/tags` - создание пользовательского тега.
- `PATCH /api/v1/tags/:id` - обновление пользовательского тега.
- `DELETE /api/v1/tags/:id` - удаление пользовательского тега.

Поддерживаются два типа тегов:

- Пользовательские теги, принадлежащие конкретному пользователю.
- Системные теги, доступные всем пользователям.

Системные теги ограничены именами:

- `reporting`
- `operations`
- `call`

Системные теги нельзя изменять или удалять через API. Имена тегов уникальны без учета регистра в рамках своего scope: среди системных тегов или среди тегов одного пользователя.

### Связь задач и тегов

Реализована привязка тегов к задачам текущего пользователя:

- `POST /api/v1/tasks/:task_id/tags` - добавить тег к задаче.
- `DELETE /api/v1/tasks/:task_id/tags/:tag_id` - убрать тег с задачи.

К задаче можно привязать пользовательский тег текущего пользователя или системный тег. Чужие задачи и чужие пользовательские теги недоступны.

### Настройки периодичности задач

Реализованы настройки повторения задач:

- ежедневные задачи с интервалом `n` дней;
- ежемесячные задачи на конкретное число месяца;
- задачи на конкретные даты;
- задачи по чётным дням месяца;
- задачи по нечётным дням месяца.

Список задач за период строится через сервис `Tasks::IndexQuery`. Для повторяющихся задач сервис использует `Tasks::RecurrenceDatesBuilder`, который вычисляет даты выпадений внутри запрошенного временного окна.

Для формирования элемента ответа используется `Tasks::OccurrenceBuilder`, который подставляет индивидуальный статус из `TaskOccurrence`, если он существует, либо базовый статус задачи.

Изменение статуса конкретного экземпляра реализовано через сервис `Tasks::OccurrenceStatusUpdater`.

### Swagger

OpenAPI-документация сгенерирована через rswag:

- Swagger UI: `/api-docs`
- OpenAPI YAML: `/api-docs/v1/swagger.yaml`
- Сгенерированный файл: `swagger/v1/swagger.yaml`

Документация покрывает аутентификацию, задачи, повторяющиеся задачи, теги и привязку тегов к задачам.

## Тесты

В проекте используется RSpec.

Покрытие request specs:

- Регистрация пользователя, ошибки регистрации.
- Вход пользователя, неверные учетные данные.
- Выход пользователя и отзыв JWT.
- Защита эндпоинтов без JWT.
- CRUD задач.
- Изоляция задач по текущему пользователю.
- Фильтрация задач по статусу, точной дате и диапазону дат.
- Получение повторяющихся задач в списке задач за период.
- Проверка, что виртуальные экземпляры повторяющихся задач не создают новые записи в `tasks`.
- Обновление статуса конкретного экземпляра повторяющейся задачи.
- Проверка, что изменение одного экземпляра не меняет статус всей серии.
- CRUD тегов.
- Доступность системных тегов.
- Запрет изменения и удаления системных тегов.
- Изоляция пользовательских тегов между пользователями.
- Привязка пользовательских и системных тегов к задачам.
- Повторная привязка существующего тега без дублирования.
- Удаление связи задачи и тега.
- OpenAPI/Swagger request specs для генерации `swagger/v1/swagger.yaml`.

Покрытие model specs:

- `User`: Devise-модули, обязательные поля, case-insensitive уникальность email, удаление зависимых задач.
- `Task`: список допустимых статусов, обязательные поля, принадлежность пользователю, запрет неизвестных статусов, валидация настроек повторения.
- `TaskOccurrence`: обязательные поля, допустимые статусы, уникальность пары `task_id + occurrence_date`.
- `JwtDenylist`: spec-файл создан, поведенческие проверки не добавлены.

Покрытие service specs:

- `Tasks::RecurrenceDatesBuilder` - генерация дат повторений для `daily`, `monthly_day`, `specific_dates`, `even_days`, `odd_days`.
- `Tasks::IndexQuery` - формирование календарной выдачи обычных и повторяющихся задач.
- `Tasks::OccurrenceBuilder` - построение элемента выдачи с учетом индивидуального статуса экземпляра.
- `Tasks::OccurrenceStatusUpdater` - создание и обновление статуса конкретного экземпляра повторяющейся задачи.

Текущая проверка:

```bash
bundle exec rspec
```

Перед отправкой тестового задания необходимо выполнить полный прогон тестов и убедиться, что все specs проходят.

## Допущения и ограничения

- Авторизация реализована через Devise JWT, хотя в исходном задании она не была обязательной. Это добавлено как улучшение API.
- Каждая задача принадлежит конкретному пользователю.
- Пользователь видит и изменяет только свои задачи.
- Системные теги являются глобальными и доступны всем пользователям.
- Пользовательские теги принадлежат конкретному пользователю.
- Повторяющаяся задача хранится как одна запись в `tasks`, а не как набор заранее созданных задач.
- Для повторяющихся задач `due_date` используется как базовая дата задачи.
- Если для повторяющейся задачи не передан `recurrence_starts_on`, он автоматически устанавливается равным `due_date`.
- Если `recurrence_ends_on` равен `null`, серия считается не имеющей даты окончания.
- API не генерирует экземпляры повторяющихся задач без ограниченного временного окна. По умолчанию используется окно от текущей даты на 30 дней.
- Максимальный диапазон `date_from` / `date_to` ограничен 366 днями.
- Для `monthly_day` со значением `31` месяцы без 31-го числа пропускаются.
- Для `specific_dates` даты должны быть переданы в формате ISO8601.
- Weekly recurrence не реализован, потому что он не указан в обязательном списке типов повторения. Текущая архитектура позволяет добавить его через новый `recurrence_type`.
- Перенос или отмена одного экземпляра задачи как отдельное исключение из правила не реализованы, так как это бонусная часть задания. Архитектурно это можно добавить через отдельную таблицу occurrence overrides.

## Локальный запуск

### Без Docker

```bash
bundle install
bin/setup --skip-server
bin/rails db:prepare
bin/rails server
```

По умолчанию используются параметры БД:

- `DATABASE_HOST=localhost`
- `DATABASE_USERNAME=postgres`
- `DATABASE_PASSWORD=postgres`
- `DATABASE_NAME=medods_tracker_api_development`
- `DATABASE_NAME_TEST=medods_tracker_api_test`

### Через Docker Compose

```bash
docker compose up --build
```

Приложение будет доступно на `http://localhost:3000`, PostgreSQL - на `localhost:5432`.

## Полезные команды

```bash
bin/setup                         # установка зависимостей и подготовка проекта
bin/rails db:prepare              # создание и миграция БД
bundle exec rspec                 # запуск тестов
bundle exec rails rswag:specs:swaggerize
bin/rubocop                       # проверка Ruby-стиля
bin/brakeman                      # security scan Rails-кода
bin/bundler-audit                 # проверка гемов на известные уязвимости
bin/ci                            # локальный CI-пайплайн
docker compose up --build         # запуск приложения и PostgreSQL в контейнерах
```
