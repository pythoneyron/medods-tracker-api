## 14. Реализация повторяющихся задач: архитектурное решение

### Запрос

```text
Предложить лучшее решение для повторяющихся задач.

Нужно поддержать:

- ежедневные задачи каждый n-й день;
- ежемесячные задачи на конкретное число месяца;
- задачи на конкретные даты;
- задачи по чётным/нечётным дням месяца.

Также нужно решить две ключевые проблемы:

1. Не создавать бесконечное количество задач на годы вперёд.
2. Дать каждому экземпляру повторяющейся задачи независимый status.
```

### Результат

```text
Была выбрана гибридная архитектура:

1. Повторяющаяся задача хранится как одна запись в таблице tasks.
2. Правило повторения хранится в полях:
   - recurrence_type
   - recurrence_config
   - recurrence_starts_on
   - recurrence_ends_on
3. Экземпляры повторяющихся задач не создаются заранее.
4. Для GET /api/v1/tasks экземпляры генерируются только внутри запрошенного временного окна.
5. Индивидуальное состояние конкретного экземпляра хранится в task_occurrences.
6. Изменение статуса одного экземпляра не меняет status всей серии.
```

### Architectural decision

```text
Для решения проблемы бесконечности система не создаёт задачи на будущие даты. Вместо этого она хранит одну задачу-серию и генерирует виртуальные экземпляры только на запрошенный период.

Для решения проблемы независимого статуса используется таблица task_occurrences. Запись создаётся только тогда, когда пользователь меняет статус конкретного экземпляра. Если записи нет, экземпляр использует базовый status из tasks.
```
