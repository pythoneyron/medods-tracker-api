# Medods Tracker API

API-only приложение на Ruby on Rails для модуля трекера рабочих задач в медицинской информационной системе. Сервис позволяет персоналу регистрироваться, авторизоваться по JWT, вести личные задачи, фильтровать их по статусу и дате, а также помечать задачи пользовательскими и системными тегами.

## Технологии

- Ruby `4.0.1`
- Ruby on Rails `8.1.3` в режиме API-only
- PostgreSQL
- Puma
- Jbuilder для JSON-ответов
- Devise и `devise-jwt` для регистрации, входа, выхода и JWT-аутентификации
- `JwtDenylist` для ревокации JWT при выходе
- RSpec, FactoryBot, Shoulda Matchers и Faker для тестов
- rswag для OpenAPI/Swagger-документации
- RuboCop, Brakeman и Bundler Audit для статических проверок
- Docker и Docker Compose для контейнерного запуска

## Реализованный функционал

### Аутентификация

Все пользовательские API-эндпоинты находятся под префиксом `/api/v1`.

- `POST /api/v1/users` - регистрация пользователя.
- `POST /api/v1/users/sign_in` - вход пользователя.
- `DELETE /api/v1/users/sign_out` - выход пользователя и отзыв текущего JWT.
- JWT возвращается в заголовке `Authorization`.
- Защищенные эндпоинты требуют заголовок `Authorization: Bearer <token>`.

### Задачи

Реализован базовый CRUD задач:

- `GET /api/v1/tasks` - список задач текущего пользователя.
- `GET /api/v1/tasks/:id` - просмотр одной задачи текущего пользователя.
- `POST /api/v1/tasks` - создание задачи для текущего пользователя.
- `PATCH /api/v1/tasks/:id` - обновление задачи текущего пользователя.
- `DELETE /api/v1/tasks/:id` - удаление задачи текущего пользователя.

Задача содержит:

- `title` - название.
- `description` - описание.
- `due_date` - дата выполнения.
- `status` - статус.
- `tags` - связанные теги.

Поддерживаемые статусы:

- `new`
- `pending`
- `in_progress`
- `done`
- `cancelled`

Для списка задач реализованы фильтры:

- `status` - фильтр по статусу.
- `date` - фильтр по точной дате выполнения.
- `date_from` и `date_to` - фильтр по диапазону дат.

Задачи изолированы по пользователям: пользователь не может получить, изменить или удалить чужую задачу.

### Теги

Реализован CRUD тегов:

- `GET /api/v1/tags` - список доступных тегов.
- `GET /api/v1/tags/:id` - просмотр тега.
- `POST /api/v1/tags` - создание пользовательского тега.
- `PATCH /api/v1/tags/:id` - обновление пользовательского тега.
- `DELETE /api/v1/tags/:id` - удаление пользовательского тега.

Поддерживаются два типа тегов:

- Пользовательские теги, принадлежащие конкретному пользователю.
- Системные теги, доступные всем пользователям.

Системные теги ограничены именами:

- `reporting`
- `operations`
- `call`

Системные теги нельзя изменять или удалять через API. Имена тегов уникальны без учета регистра в рамках своего scope: среди системных тегов или среди тегов одного пользователя.

### Связь задач и тегов

Реализована привязка тегов к задачам текущего пользователя:

- `POST /api/v1/tasks/:task_id/tags` - добавить тег к задаче.
- `DELETE /api/v1/tasks/:task_id/tags/:tag_id` - убрать тег с задачи.

К задаче можно привязать пользовательский тег текущего пользователя или системный тег. Чужие задачи и чужие пользовательские теги недоступны.

### Swagger

OpenAPI-документация сгенерирована через rswag:

- Swagger UI: `/api-docs`
- OpenAPI YAML: `/api-docs/v1/swagger.yaml`
- Сгенерированный файл: `swagger/v1/swagger.yaml`

Документация покрывает аутентификацию, задачи, теги и привязку тегов к задачам.

## Тесты

В проекте используется RSpec.

Покрытие request specs:

- Регистрация пользователя, ошибки регистрации.
- Вход пользователя, неверные учетные данные.
- Выход пользователя и отзыв JWT.
- Защита эндпоинтов без JWT.
- CRUD задач.
- Изоляция задач по текущему пользователю.
- Фильтрация задач по статусу, точной дате и диапазону дат.
- CRUD тегов.
- Доступность системных тегов.
- Запрет изменения и удаления системных тегов.
- Изоляция пользовательских тегов между пользователями.
- Привязка пользовательских и системных тегов к задачам.
- Повторная привязка существующего тега без дублирования.
- Удаление связи задачи и тега.
- OpenAPI/Swagger request specs для генерации `swagger/v1/swagger.yaml`.

Покрытие model specs:

- `User`: Devise-модули, обязательные поля, case-insensitive уникальность email, удаление зависимых задач.
- `Task`: список допустимых статусов, обязательные поля, принадлежность пользователю, запрет неизвестных статусов.
- `JwtDenylist`: spec-файл создан, поведенческие проверки не добавлены.

Текущая проверка:

```bash
bundle exec rspec
```

Результат последнего запуска:

```text
91 examples, 0 failures
```

## Локальный запуск

### Без Docker

```bash
bundle install
bin/setup --skip-server
bin/rails db:prepare
bin/rails server
```

По умолчанию используются параметры БД:

- `DATABASE_HOST=localhost`
- `DATABASE_USERNAME=postgres`
- `DATABASE_PASSWORD=postgres`
- `DATABASE_NAME=medods_tracker_api_development`
- `DATABASE_NAME_TEST=medods_tracker_api_test`

### Через Docker Compose

```bash
docker compose up --build
```

Приложение будет доступно на `http://localhost:3000`, PostgreSQL - на `localhost:5432`.

## Полезные команды

```bash
bin/setup                         # установка зависимостей и подготовка проекта
bin/rails db:prepare              # создание и миграция БД
bundle exec rspec                 # запуск тестов
bundle exec rails rswag:specs:swaggerize
bin/rubocop                       # проверка Ruby-стиля
bin/brakeman                      # security scan Rails-кода
bin/bundler-audit                 # проверка гемов на известные уязвимости
bin/ci                            # локальный CI-пайплайн
```
